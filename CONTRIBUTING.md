# Contributing

This repository uses a dual-layer structure (Core chapters + Bench Notes) with strict conventions for reproducibility and citeability. Read this before writing content.

---

## The two content types

### Core chapters (`parts/`)

Chapters follow the **depth ladder** — three tiers, each as a distinct callout block:

| Tier | Callout | Audience | Rough length |
|------|---------|----------|-------------|
| **Core** | `{.callout-note}` | Third-year undergraduate and above | 3–6 pages |
| **Going deeper** | `{.callout-tip}` | MSc/PhD | 1–4 pages |
| **Bench** | `{.callout-warning}` | All — signposts to Bench Notes | ½ page |
| **Key references** | `{.callout-important}` | All | ½ page |

Copy `parts/_chapter-template.qmd` and fill in every section. Do not skip tiers.

**Maths policy:** derivations belong in *Going deeper*, not Core. Core states results and builds intuition. Math is unbounded once you are in the *Going deeper* tier.

**Citation policy:** every factual claim that can be cited must be cited with `[@key]`. Citation keys live in `references.bib`. If the source is not already there, add a complete BibTeX entry; flag uncertain fields with `%% VERIFY:` comments.

### Bench Notes (`bench-notes/`)

A Bench Note has exactly **six fixed fields** — the template enforces them:

1. **Concept** — one paragraph, why it matters experimentally.
2. **Governing equations** — the key formulae, with a symbol table.
3. **Decision rule / table** — when to use what; prefer a table.
4. **Failure modes** — symptom → cause → fix.
5. **See also** — cross-references to Core modules and related Bench Notes.
6. **References** — `[@key; @key2]`.

The Bench Note format is intentionally dense. No prose padding. If you need a third paragraph, it probably belongs in a Core chapter, not a Bench Note.

After writing a Bench Note:
- Add a card link in `bench-notes/index.qmd` under the appropriate group.
- Add the file to `_quarto.yml` under `part: "Bench Notes"`.

## Reproducible figures

**Every figure must be reproducible from the repo alone.** That means:

1. The figure is generated by code — either a script in `figures/src/` or an executable code cell in the `.qmd` file.
2. All data the code reads is in `data/` (or simulated with a fixed seed).
3. Random seeds are set explicitly: `rng = np.random.default_rng(seed=42)`.
4. The figure output (`.pdf`, `.png`) is in `.gitignore` — it is a *build artifact*, not a tracked file.
5. The script works standalone: `python figures/src/my_figure.py` must run cleanly in the repo's venv.

**Style:** apply `plt.rcParams.update({...})` at the top of every script. Copy the style block from `figures/src/chi2_fit_residuals.py`. This keeps all figures visually consistent.

**No manual screenshots.** No figures that require external tools, proprietary software, or undocumented steps.

## Citation style

- Format: APS (American Physical Society) style via `csl/american-physics-society.csl`.
- Cite with `[@key]` for inline, `[@key, p. 42]` for a specific page.
- Every BibTeX entry in `references.bib` must include author, title, publisher, year, and (if available) ISBN/DOI/URL. Flag uncertain fields with `%% VERIFY:`.
- Do not create aliases or shorthand keys — use the full canonical key (e.g. `horowitzHill2015`).

## Cross-references

Use Quarto's native cross-reference syntax throughout:

```
@fig-<label>         # figure
@tbl-<label>         # table
@eq-<label>          # equation
@sec-<label>         # section
```

Every equation that might be referenced should have a `{#eq-<slug>}` label. Every section in a Core chapter should have a `{#sec-<slug>}` label. Never use bare `(1)` or `Figure 3` in running text.

## Quarto version

The project is pinned to Quarto **1.5.57** in `.github/workflows/publish.yml`. If you upgrade, update that pin explicitly and test both `--to html` and `--to pdf` before merging.
